## 125A-5 — Multi-proyecto: soporte completo en chatbot WhatsApp
- **Que:** Investigación completada + implementación completa del manejo multi-proyecto en el chatbot. El chatbot ahora puede: (1) listar proyectos disponibles desde `config/opencode-projects.json`, (2) cambiar el proyecto activo por usuario (persistido en user_meta), (3) listar ramas de cualquier proyecto vía consulta rápida (git branch -a ejecutado por el runner sin abrir OpenCode).
  - `AgentChatProcessor.php`: 4 nuevos métodos auxiliares (loadProjectsConfig, getActiveProject, setActiveProject, getActiveProjectBranch), 3 nuevas acciones en el switch (listar_proyectos, listar_ramas, cambiar_proyecto), sistema prompt actualizado con las 3 acciones + reglas, buildContexto() ahora muestra proyecto activo.
  - `opencode-whatsapp-runner.mjs`: handleQuickCommand() ejecuta comandos git vía spawnSync para consultas rápidas sin lanzar OpenCode; pollOnce() detecta es_consulta_rapida en payload y deriva a quick command.
  - `AgentRestHandlers.php`: reportarOpencodeJob() muestra iconos 🔍/❌ para consultas rápidas en lugar de iconos de OpenCode.
- **Archivos:** `App/Services/AgentChatProcessor.php`, `scripts/opencode-whatsapp-runner.mjs`, `App/Services/AgentRestHandlers.php`
- **Gotchas:** El listado de ramas se ejecuta en el directorio del proyecto usando spawnSync con shell=true; esto requiere que el proyecto tenga git instalado y el directorio exista localmente. Para proyectos no clonados localmente, listar_ramas fallará con error.
- **Sentinel:** no aplica (cambio funcional, no estructural).
- **GLORY:** No aplica (integración específica del chatbot WhatsApp).

## 126A-1 — Fix: editar_recordatorio no soportaba cambiar recurrence_minutes
- **Que:** El handler `editar_recordatorio` ignoraba el parámetro `recurrence_minutes`, por lo que cambiar el intervalo de un recordatorio (ej: de 30 a 15 min) no surtía efecto. El LLM recibía `exito: true` pero el payload quedaba con el intervalo viejo, haciendo que la siguiente recurrencia se programara con el intervalo incorrecto.
  - Se agregó `recurrence_minutes` al system prompt del LLM en la definición de `editar_recordatorio`.
  - Se agregó recolección de `recurrence_minutes` en el handler, mergándolo al payload via `payload_merge`.
- **Archivos:** `App/Services/AgentChatProcessor.php`
- **Gotchas:** El cambio solo afecta a la acción pendiente actual. Si ya hay ocurrencias futuras creadas, no se actualizan retroactivamente (pero al ejecutarse la siguiente, usará el nuevo intervalo automáticamente).
- **Sentinel:** no aplica (el bug era lógico, no estructural).
- **GLORY:** No aplica (es específico del scheduler del agente).
