# Tareas completadas — 2026-03-30

## 303A-9 — Fix CSP: eliminar inline handlers de la extensión fb-group-manager
- **Que:** La extensión de Chrome no mostraba grupos porque MV3 bloquea inline event handlers (CSP `script-src 'self'`). Se eliminaron 6 inline handlers en total: 2x `onclick="event.stopPropagation()"` en links de grupo (dashboard.ts), 3x `onerror="this.style.display='none'"` en avatares (dashboard.ts), 1x `onerror` complejo en popup.ts (con fallback a placeholder). Todos reemplazados por event listeners JS usando `addEventListener` después de inyectar innerHTML.
- **Archivos:**
  - `Extensión/fb-group-manager/src/dashboard/dashboard.ts` — Eliminados 5 inline handlers de template literals. Añadidos 3 bloques de event delegation: `.group-link-stop` click (stopPropagation), `.group-table-avatar` error (hide), `.group-detail-avatar` error (hide en modal).
  - `Extensión/fb-group-manager/src/popup/popup.ts` — Eliminado 1 `onerror` inline en avatar. Añadido listener `.group-avatar` error con fallback a placeholder.
- **Gotchas:** El proyecto Extensión no tiene repo git, así que no hay commit. El build (`node build.js`) compila TS a dist/ correctamente. Code Sentinel reporta innerHTML warnings pre-existentes (el HTML usa `escapeHtml()` para sanitizar).
- **Sentinel:** No aplica (proyecto externo, no glorytemplate).

## 303A-10 — Fix lifecycle: extensión deja de funcionar al cerrar/reabrir pestaña Facebook
- **Que:** El content script se "moría" permanentemente cuando el service worker de MV3 se dormía y acumulaba MAX_FAILURES fallos de mensaje. Una vez que `contextInvalidated=true` y `cleanup()` corría, no había forma de recuperarse — el observer, el interval y el scan morían para siempre. Además, Facebook usa SPA navigation (History API) que el content script no interceptaba, causando que las navegaciones intra-Facebook no dispararan nuevos scans.
- **Archivos:**
  - `Extensión/fb-group-manager/src/content-scripts/group-detector.ts` — (1) `recordMessageFailure`: si `chrome.runtime.id` sigue existiendo tras MAX_FAILURES, ya no invalida permanentemente; en su lugar llama `scheduleReconnect()` que espera 5s y reinicializa observer+interval+scan. (2) Nueva función `hookSpaNavigation()` que monkey-patches `history.pushState`/`replaceState` y escucha `popstate` para escanear en cada navegación SPA. (3) Nuevo handler `PING` en `onMessage` que revive el content script si el background le hace ping y el contexto estaba muerto pero `chrome.runtime.id` sigue vivo.
  - `Extensión/fb-group-manager/src/background.ts` — Añadido `chrome.tabs.onUpdated` listener que envía `PING` al content script cuando una pestaña de Facebook con URL `/groups` completa la carga. Esto cubre: recargar pestaña, navegar de vuelta a FB, despertar el tab.
- **Gotchas:** 
  - Si la extensión realmente se desinstala/recarga (`chrome.runtime.id` deja de existir), `contextInvalidated=true` sigue siendo permanente (correcto).
  - El monkey-patch de `history.pushState/replaceState` solo afecta la instancia actual de la página; si se recarga la pestaña, Chrome re-inyecta el content script y se vuelve a hacer el patch.
  - El PING del background usa `.catch()` silencioso porque puede llegar antes de que el content script se haya inyectado.
- **Sentinel:** No aplica (proyecto externo).

## 303A-11 — Auditoría chatbot: confirmación obligatoria para eliminaciones
- **Que:** Auditoría completa del panel IA (chatbot). Se encontró una vulnerabilidad crítica: el LLM podía ejecutar `eliminar_tarea` y `eliminar_habito` sin confirmación del usuario. Un prompt ambiguo ("limpia mi dashboard") podía resultar en eliminaciones irreversibles. Se implementó un sistema de confirmación: las acciones destructivas ahora se marcan como `pendienteConfirmacion: true` y el usuario debe hacer click en "Confirmar" o "Cancelar" antes de que se ejecuten.
- **Archivos:**
  - `App/React/stores/iaStore.ts` — Añadido campo `pendienteConfirmacion?: boolean` a `AccionIA`. Añadida acción `actualizarMensaje(id, cambios)` para modificar mensajes existentes.
  - `App/React/config/accionesIA.ts` — `eliminar_tarea` y `eliminar_habito` ya no ejecutan la eliminación directamente; marcan `pendienteConfirmacion: true` con descripción del ítem. Añadida función `ejecutarAccionDestructiva()` para cuando el usuario confirma. System prompt mejorado: instruye al LLM a no eliminar sin petición explícita y advierte que las eliminaciones requieren confirmación UI.
  - `App/React/services/iaService.ts` — Propaga `pendienteConfirmacion` de ResultadoAccion a AccionIA.
  - `App/React/hooks/paneles/usePanelIA.ts` — Nuevas funciones `confirmarAccion(mensajeId, indice)` y `rechazarAccion(mensajeId, indice)`. Confirmar ejecuta la eliminación real via `ejecutarAccionDestructiva`. Rechazar marca como cancelado.
  - `App/React/components/paneles/PanelIA.tsx` — Acciones pendientes se renderizan con icono AlertTriangle amarillo + botones "Confirmar" (rojo) y "Cancelar" (neutro) en vez del badge simple de éxito/error.
  - `App/React/styles/dashboard/componentes/panelIA.css` — Nuevo estilo `.panelIAAccionBadge--pendiente` (fondo amarillo), `.panelIAAccionBtn` con variantes `--confirmar` (rojo) y `--cancelar` (neutro).
- **Gotchas:**
  - Los `<button>` nativos dentro del badge son intencionales (Code Sentinel sugiere `<Boton>` pero para micro-botones inline el componente genérico es overkill).
  - El system prompt NO elimina las acciones `eliminar_tarea/eliminar_habito` del schema — las mantiene para que el LLM pueda proponer eliminaciones, pero ahora siempre requieren confirmación técnica.
  - Si `chrome.runtime.id` deja de existir, `contextInvalidated=true` sigue siendo permanente (no afecta al chatbot, es de la extensión).
- **Sentinel:** Falso positivo: Sentinel reporta "usar <Boton>" para los micro-botones inline de confirmación. Son buttons de 10px dentro de un badge — no aplica el componente genérico.
